좀 더 설명을 드리면,
콘솔에서 로그인을 위해서는 ttyX.service가 실행되어 있으며, 로그아웃시에 해당 서비스를 Restart하게 됩니다. 해당 로그는 이때('Restart') Info성 로그를 출력하게 됩니다.

static int service_dispatch_timer(sd_event_source *source, usec_t usec, void *userdata)
{
..
case SERVICE_AUTO_RESTART:
log_unit_info(UNIT(s)->id,
s->restart_usec > 0 ?
"%s holdoff time over, scheduling restart." :
"%s has no holdoff time, scheduling restart.",
UNIT(s)->id);
service_enter_restart(s);
break;
..
}
======================
즉, 해당 로그는 시스템 운영에 있어서 어떤 문제의 로그는 아닙니다.


만약 해당 로그를 나타내지 않게 하기 위해서는 일전에 케이스(#01573782)를 통해서 안내드렸던 방법으로 제거하실 수 있습니다.

for sample, (이전 케이스 예시)
echo 'if $programname == "systemd" and ($msg startswith "Starting Session" or $msg startswith "Started Session" or $msg startswith "Created slice" or $msg startswith "Starting user-") then stop' >/etc/rsyslog.d/ignore-systemd-session-slice.conf

안녕하세요? Red Hat 송창안 입니다.

문의 주신 내용은 기본적으 systemd에서 제공하는 메세지이며, 서비스이름 holdoff time over, scheduling restart을 메세지를 정보를 출력 하게 됩니다.
systemd 의 이슈 보다는 getty 서비스에 확인이 필요한것으로 사료 됩니다.

getty 관련 서비스 설정 파일 위치는 아래와 같으며,
/etc/systemd/system/getty.target.wants/getty\@tty1.service

현재의 이슈를 확인 하기 위해서 기본적으로 확인이 몇가지 정보가 필요하며,

1. RHEL7에서의 X서버 사용 유무, /var/log/Xorg.*.log
2. 이슈 발생시 , journalctl -b | grep -i getty
3. /var/log/message
4. /var/log/secure
5. sosreport

정보를 제공 해주시면 로그 및 내용을 파악 후, 다시 업데이트 하도록 하겠습니다.
