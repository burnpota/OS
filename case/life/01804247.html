안녕하세요

Red Hat Global Support Services 를 이용해 주셔서 감사합니다.

kdump 설정에 vmcore 가 vg0 볼륨장치에 DUMP 장치에 생성되도록 설정되어있습니다.

[/etc/kdump.conf]
raw /dev/vg0/DUMP
core_collector makedumpfile -c -d 17

[lsblk]
`-sda6 8:6 0 211.9G 0 part
|-vg0-sysadmin (dm-3) 253:3 0 10G 0 lvm /sysadmin
|-vg0-home (dm-4) 253:4 0 10G 0 lvm /home
|-vg0-CRASH (dm-5) 253:5 0 70G 0 lvm /CRASH
`-vg0-DUMP (dm-6) 253:6 0 70G 0 lvm

아래 로그를 마지막으로 시스템이 행상태에 빠진 것으로 추정하고 있습니다.
하지만, 로그 자체에서는 시스템 행에 대한 원인을 확인할 수 없었습니다.
현재 남아있는 로그를 가지고서는 시스템 행에 대한 원인을 확인하기 힘든 상황입니다.

Feb 26 20:36:01 SLPSADEL01 CROND[8597]: (simsplk1) CMD (python /APP/splunkLog/script/logCheck.py resetting)
Feb 26 20:36:01 SLPSADEL01 CROND[8598]: (simsplk1) CMD (/APP/splunk/splunk_start.sh)
Feb 26 20:36:01 SLPSADEL01 CROND[8599]: (root) CMD (/usr/lib64/sa/sa1 1 1)
Feb 26 20:36:01 SLPSADEL01 CROND[8594]: (root) MAIL (mailed 67 bytes of output but got status 0x004b#012)

만일 동일한 증상 발생시에는 강제 리부팅보다는 매직키를 이용하여 vmcore 를 수집해주시거나
하드웨어 NMI 버튼이 존재한다면 해당 버튼을 눌러서 vmcore 를 수집해주시면 시스템 행에 대한
추가적인 분석이 가능합니다.

콘솔에 프롬프트가 멈춰있을 경우에 매직 키 활성화가 되어있다면,
Alt+PrintScreen+[C] 키를 콘솔에서 입력하여 강제 패닉을 발생할 수 있습니다.

* What is the SysRq Facility and how do I use it?
https://access.redhat.com/articles/231663

또한, kdump 가 아래 상황에서 동작할 수 있게 다음 설정을 해주시기 바랍니다.

* What panic parameters are available for use to panic a system when it is hanging or sluggish?
https://access.redhat.com/solutions/459523
