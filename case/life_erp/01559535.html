제목 : SWAP 권고 사이즈

======================
사전문의

어떤 문제/오류/결함이 발생했습니까? 기대하시는 결과는 무엇입니까?

현재 보험 ERP 구축 예정인 non-sap인 장비의 메모리 사이즈가 96 /120 / 196 / 384 GB 로 64GB를 모두 넘어가는 서버입니다.

기존 권고 Swap Size 테이블을 보면 아래와 같습니다.

============================================================
RAM 2GB 작은 경우 → RAM의 2배
RAM 2GB ~ 8GB → RAM 동일 사이즈
RAM 8GB ~ 64GB → RAM의 0.5배
RAM 64GB 클 경우 → 업무 고려한 설정
============================================================

SAP 시스템들은 SAP 의 권고 값으로 설정하기 때문에 문제가 없으나,

Non-SAP 의 경우 64GB 를 모두 넘어가므로 적당한 사이즈를 업무를 고려한 설정으로 잡아야 할 것 같습니다.

업무를 고려한 설정에 대한 적당한 계산 방법 또는 기준 값을 알고 싶습니다.

안녕하세요,
Red Hat 한진구 입니다.

우선 RHEL7 문서와 저희 KCS 문서상의 수치상의 조금 차이가 있어서 혼란을 드리는 것 같아서 죄송합니다.

RHEL7: https://access.redhat.com/documentation/en-US/Red_Hat_Enterprise_Linux/7/pdf/Installation_Guide/Red_Hat_Enterprise_Linux-7-Installation_Guide-en-US.pdf
RH KCS: https://access.redhat.com/solutions/15244


정리를 해보면, 8-64GB 아상의 메모리를 가진 시스템에서 SWAP의 사이즈는 workload 에 따라 조금의 차이는 있을 수 있겠으나, 최소 4GB이상으로 메모리의 0.5배수를 해서 SWAP값을 잡으면 좋을 것 같습니다. 다만, 여유의 공간이 없을 경우에 최소 4GB를 설정하라고 보시면 될 것 같습니다.


또한, 64GB 이상의 메모리를 가진 시스템의 경우에는 최소 4GB는 주되, SWAP 공간의 사이즈를 0.5배 할 경우 메모리 사이즈에 따라 너무 큰 공간이 될 수도 있기 때문에 workload에 따른 적절한 수치를 정해야 합니다.
예를들어, 384GB의 경우에는 0.5배수가 벌써 192GB가 되기 때문에 이 공간이 비용적으로 무시할 수 있는 공간이라고 보기 어려울 것 같습니다. 그러다보니 일반적인 방법론을 찾아야 할 것으로 보이며, 저는 이에 대해서 아래와 같은 방법을 제안해드립니다.

8-64GB: 0.5배수
64GB이상의 경우, 32GB + 64GB를 초과하는 메모리양의 ROOT(제곱근) 산술식

예를들어, 384GB = 64GB+320GB => SWAP: 64 * 0.5GB + 32 제곱근 'ROOT (32)' = 32 + 17GB = 49GB
