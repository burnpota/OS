첨부 해 주신 내용에 대해 아래와 같이 간단하게 설명 해 드립니다.

현재 발견 된 취약점은 사용자 공간 바이너리 스택에 할당되는 메모리 공간에서 발생 한 것입니다.
합 (또는 다른 메모리 영역)과 스택 메모리 영역이 서로 인접 해있는 경우 공격자는 이 취약점을
이용하여 스택 가드 갭을 뛰어넘어 프로세스 스택이나 인접한 메모리 영역에서 메모리 손상을 일으켜
공격자의 권한을 증가시킬 수 있습니다.

해당 취약점에 대해 스택 가드 갭 크기를 1 MiB로 늘려서 성공적으로 약용하는 것을 어렵게
만들었습니다. 기존 가드 갭 크기는 one page(4KiB)였습니다.

해당 취약점 문제는 현재 ongoing 상태로 되어 있으나 RHEL7.2 EUS 또한 RHEL6.7 EUS
버전에서는 패치가 되어 있는 상태이며 3.10.0-327.55.2.el7 또는 2.6.32-573.42.2.el6
로 커널 업데이트 하시면 해당 이슈 회피 하실 수 있습니다.

https://access.redhat.com/security/cve/CVE-2017-1000364

안녕하세요?
A flaw was found in the way memory was being allocated on the stack for user space binaries. If heap (or different memory region) and stack memory regions were adjacent to each other, an attacker could use this flaw to jump over the stack guard gap, cause controlled memory corruption on process stack or the adjacent memory region, and thus increase their privileges on the system. This is a kernel-side mitigation which increases the stack guard gap size from one page to 1 MiB to make successful exploitation of this issue more difficult.

위 내용을 간단하게 설명 부탁드리고
guard gap이 1Mib라고 했는데 기존은 얼마 였는지 궁금합니다.

그리고 말씀중인 ongoing상태는 에라타에 참조에 링크가 걸려 있씁니다.

For more information on the kernel & glibc Stack Guard security vulnerability ( CVE-2017-1000364, CVE-2017-1000366, CVE-2017-1000367) please refer to this Vulnerability Page.

이 링크를 가보시면 ongoing 상태입니다.
https://access.redhat.com/security/vulnerabilities/stackguard
